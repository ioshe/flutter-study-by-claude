# 2026-05-28 — Flutter 자체 렌더링(게임 엔진 접근)의 장점과 단점 (심화)

> 📌 **추가학습 출처**: `20260528_flutter_engine_구조.md` (핵심 정의 / 왜 등장했는가)
> 원문: *"Flutter 의 가장 급진적인 결정: OS 의 네이티브 위젯을 안 쓴다. OS 한테서 빈 캔버스 한 장만 받아 그 위의 모든 픽셀을 직접 그린다. 게임 엔진과 같은 접근."*
> 이 문서는 그 "게임 엔진식 접근" 이 가져온 **장점과 단점**을 깊게 파고든다.

---

## 핵심 정의

**Flutter 는 OS 가 제공하는 UI 부품(버튼, 텍스트박스, 스크롤뷰 등)을 일절 쓰지 않고, OS 에게서는 "그림 그릴 빈 캔버스(surface)" 하나만 받아, 그 위의 모든 픽셀을 자기 렌더링 엔진으로 직접 그린다.**

이걸 게임에 비유하는 이유: 게임(예: 유니티로 만든 게임)도 OS 의 버튼을 안 써. 화면 전체를 게임 엔진이 매 프레임 직접 그리잖아? Flutter 도 똑같이 **앱 전체를 매 프레임 직접 그리는 "UI 엔진"** 이야.

대비되는 접근 (다시 정리):
- **React Native**: OS 의 진짜 위젯을 JS 로 조종. "리모컨으로 OS 부품 움직이기."
- **Flutter**: OS 부품 안 쓰고 캔버스에 직접 페인팅. "백지에 그림 그리기."

---

## 왜 이 결정이 그렇게 중요한가

UI 프레임워크의 역사에서 "OS 위젯을 쓰느냐 마느냐" 는 **거의 모든 것을 결정하는 분기점**이야. 이 선택 하나에서 Flutter 의 장점도 단점도 전부 파생돼. 그래서 장단점을 이해하면 "Flutter 가 왜 이렇게 동작하지?" 하는 의문의 90%가 풀려.

---

## 장점 (직접 그려서 얻은 것)

### 1. 모든 플랫폼에서 픽셀 단위로 동일한 UI

OS 위젯을 쓰면 iOS 의 스위치와 Android 의 스위치가 다르게 생겼어. 버전마다도 달라져. 직접 그리니까 **iOS든 Android든 Windows든 웹이든 완전히 똑같이** 나와. 디자이너가 의도한 그대로. "내 폰에선 잘 됐는데 다른 폰에선 깨져요" 가 거의 없어.

### 2. 브릿지 병목이 없다 → 고성능

React Native 는 JS 세계와 네이티브 위젯 세계 사이에 **브릿지**가 있고, 복잡한 애니메이션이나 빠른 스크롤에서 이 브릿지가 병목이 돼 (초당 수천 번 메시지가 오감). Flutter 는 위젯을 직접 그리고, Dart 코드는 AOT 로 네이티브 머신코드라(컴파일 편 참고), 중간 다리가 없어서 60/120fps 를 안정적으로 뽑아.

### 3. 완전한 디자인 자유도

OS 위젯에 얽매이지 않으니 **원하는 어떤 UI 든 만들 수 있어.** 곡선 애니메이션, 커스텀 모양 버튼, 화려한 전환 효과... OS 가 제공하는 부품의 한계가 곧 한계였던 다른 프레임워크와 달라. Material Design 도, iOS 의 Cupertino 룩도, 완전 커스텀도 다 "그림" 으로 구현해.

### 4. 빠른 개발 (hot reload 와 시너지)

직접 그리는 구조 + JIT(개발 모드) 조합으로 hot reload 가 매끄러워. UI 를 고치고 1초 안에 화면에 반영돼. OS 위젯을 쓰면 네이티브 재빌드가 끼어서 이렇게 빠르기 어려워.

### 5. 버전 파편화에서 자유

"안드로이드 8 에선 이 위젯이 이렇게 동작하고 12 에선 저렇게..." 같은 OS 버전별 위젯 동작 차이를 신경 안 써도 돼. Flutter 엔진이 그리는 거라 OS 버전과 무관하게 일관돼.

---

## 단점 (직접 그려서 치른 대가)

### 1. OS 가 공짜로 주던 것을 전부 재구현해야 했다

OS 위젯에는 우리가 당연하게 여기던 기능이 내장돼 있어 — **접근성(스크린리더), 텍스트 선택/복사/붙여넣기, 텍스트 입력기(IME)**. OS 위젯을 안 쓰니 Flutter 가 이걸 **처음부터 다 만들어야 했어.** 초창기 Flutter 가 이 부분에서 약했던 이유야 (지금은 많이 성숙).

### 2. 한글 입력(IME) 같은 미묘한 문제

이건 한국 개발자에게 특히 중요해. 한글은 자음+모음을 **조합**하면서 입력되는데(ㄱ→가→각...), 이 "조합 중" 상태를 OS 입력기와 Flutter 가 주고받아야 해. OS 텍스트박스를 쓰면 공짜로 되던 게, 직접 그리는 Flutter 에선 엔진이 IME 조합 이벤트를 정확히 처리해야 해서 과거에 한글 입력 버그가 종종 있었어.

### 3. 앱 용량 증가

렌더링 엔진(Skia/Impeller)과 Dart 런타임을 앱에 통째로 포함하니, **최소 앱 크기가 커.** 빈 Flutter 앱도 수 MB. 네이티브 앱이 OS 위젯을 빌려 쓰는 것과 대조적이야.

### 4. "진짜 네이티브" 느낌과의 미묘한 간극

대부분은 구별 못 하지만, 플랫폼 고유의 미세한 동작(iOS 특유의 스크롤 바운스 물리, 시스템 폰트 렌더링 디테일, 접근성 제스처)을 100% 똑같이 흉내 내긴 어려워. Flutter 가 "거의" 똑같이 만들지만 OS 가 직접 그린 것과 1:1은 아니야.

### 5. 웹에서는 이 장점이 약해진다

engine 편에서 봤듯 웹 Flutter 는 `CanvasKit`(Skia를 wasm으로) 이나 HTML 렌더러를 써. 캔버스에 직접 그리는 방식은 웹에서 **초기 로딩이 무겁고(엔진을 다운로드해야 함), SEO·텍스트 선택·접근성이 일반 HTML 사이트보다 불리해.** 그래서 Flutter 는 모바일에서 가장 강하고 웹에선 상대적으로 약해.

### 6. OS 신기능 즉시 반영 지연

OS 가 새 위젯 스타일이나 시스템 UI 를 내놓으면, OS 위젯을 쓰는 앱은 자동으로 따라가. Flutter 는 엔진/프레임워크가 그걸 "다시 그려서" 지원할 때까지 기다려야 할 수 있어.

## Flutter에서는 어떻게 체감되는가

- `MaterialApp` vs `CupertinoApp`: 둘 다 Flutter 가 "그리는" 디자인 시스템이야. iOS 에서도 Material 룩을 쓸 수 있고, Android 에서도 Cupertino(iOS) 룩을 쓸 수 있어 — OS 와 무관하게 그리니까.
- `TextField` 의 한글 입력을 테스트할 때 IME 조합이 잘 되는지 꼭 확인해야 하는 이유가 위의 단점 #2 야.
- 앱 용량을 줄이려면 `--split-per-abi`(ABI별 분리 빌드), tree shaking 등을 신경 써야 하는 이유가 단점 #3.
- 웹 배포 시 첫 로딩 스피너가 길게 느껴지는 게 단점 #5 의 결과.

## 다른 프레임워크와 비교 (철학의 스펙트럼)

```
OS 위젯 100% 의존 ←──────────────────────────→ 100% 직접 그리기
   네이티브앱        React Native       Flutter        게임엔진(Unity)
   (Swift/Kotlin)   (위젯 빌림)      (다 그림)        (다 그림)
```

- **네이티브**: OS 와 한 몸. 최고의 네이티브 경험, 하지만 플랫폼마다 따로 개발.
- **React Native**: OS 위젯 빌림. 네이티브 느낌 유지하되 브릿지 비용.
- **Flutter**: 직접 그림. 일관성·성능·자유도를 얻고 재구현·용량을 지불.
- **게임 엔진**: 극단적으로 다 그림. UI 표준 따위 없음.

**왜 Flutter 는 "직접 그리기" 극단에 가까운 선택을 했나**: 구글의 목표가 "하나의 코드로 모든 화면에서 똑같이, 빠르게" 였기 때문. 이 목표엔 OS 위젯 의존이 치명적 방해였어. 그래서 대가(접근성·IME·용량)를 감수하고 렌더링을 통째로 가져온 거야. 이건 "옳은 선택" 이라기보다 **"이 목표엔 이 선택" 이라는 트레이드오프**야.

## 주의할 점

1. **"Flutter 가 무조건 빠르고 좋다" 는 과장.** 위 단점들이 실재해. 특히 **텍스트 많은 정보성 웹사이트**라면 Flutter web 보다 일반 HTML/JS 가 거의 항상 나아.
2. **한글 IME 는 반드시 실기기에서 테스트.** 에뮬레이터·데스크톱과 모바일의 입력기 동작이 달라.
3. **접근성을 공짜로 여기지 마라.** `Semantics` 위젯 등으로 스크린리더 지원을 신경 써야 할 때가 있어 (OS 위젯이면 자동이었을 것).
4. **"네이티브처럼 보인다 ≠ 네이티브다".** 플랫폼 가이드라인을 따르고 싶으면 `Cupertino`/`Material` 을 상황에 맞게 골라 써.
