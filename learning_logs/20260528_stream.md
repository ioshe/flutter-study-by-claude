## 핵심 정의

**Stream = 시간에 걸쳐 0개 이상의 값(또는 에러)을 비동기적으로, 순서대로 내보내는 데이터의 흐름(시퀀스).**

가장 쉬운 대비:

| | Future | Stream |
|---|---|---|
| 내보내는 값 | **딱 1개** (또는 1개의 에러) | **0개 ~ 무한개** |
| 비유 | 택배 1개 (언젠가 도착) | 유튜브 생방송 (계속 흘러나옴) |
| 완료 | 값 받으면 끝 | `done` 이벤트로 끝나거나, 영원히 안 끝날 수도 |
| 받는 법 | `await` 또는 `.then()` | `await for` 또는 `.listen()` |

> 비유를 더 정확히: **Future 는 "수도꼭지에서 물 한 컵 받기"**, **Stream 은 "수도꼭지를 틀어놓고 흐르는 물을 계속 받기"** 야. 물이 언제 몇 번 나올지는 수도꼭지(=데이터 소스)가 정해.

---

## 왜 등장했는가

세상의 데이터는 두 종류야:

1. **한 번 요청하면 한 번 답이 오는 것** → 네트워크 GET 요청, 파일 한 번 읽기. → `Future` 로 충분.
2. **시간에 걸쳐 계속 발생하는 것** → 버튼 클릭(여러 번), 키보드 입력, 위치(GPS) 변화, 웹소켓 메시지, 센서 값, 타이머 틱. → `Future` 로는 표현 불가능.

`Future<int>` 는 "정수 하나가 미래에 온다" 인데, "초마다 정수가 계속 온다" 는 걸 표현할 타입이 없었어. 그래서 **Stream** 이 필요했지.

이건 사실 **Reactive Programming(반응형 프로그래밍)** 이라는 큰 패러다임의 Dart 버전이야. "데이터가 흐르고, 그 흐름에 반응해서 코드가 실행된다" 는 사고방식. JavaScript 의 RxJS, Kotlin 의 Flow, Swift 의 Combine 이 다 같은 가족이야. (뒤 '다른 언어와 비교' 에서 자세히)

> isolate 편에서 배운 **이벤트 루프** 기억나? Stream 이 내보내는 값들은 대부분 **event queue** 를 통해 전달돼. 즉 Stream 은 이벤트 루프 위에서 도는, "여러 번 발생하는 비동기 이벤트" 의 추상화야.

## Dart에서의 동작 방식

### 1) Stream 의 3가지 이벤트

Stream 은 listener 에게 3종류의 이벤트를 보내:

```
data 이벤트   → 값이 하나 도착 (여러 번 발생 가능)
error 이벤트  → 에러 발생 (스트림이 끝날 수도, 계속될 수도)
done 이벤트   → 스트림 종료 (더 이상 값 없음, 1번만)
```

### 2) 두 가지 구독 모델 (매우 중요)

| 종류 | listener 수 | 비유 | 예 |
|---|---|---|---|
| **Single-subscription** | 딱 1명만 | 1:1 전화 통화 | 파일 읽기, HTTP 응답 바디 |
| **Broadcast** | 여러 명 동시 | 라디오 방송 | 버튼 클릭, 마우스 이벤트 |

기본은 single-subscription 이야. 한 번 `listen` 하면 두 번째 `listen` 은 에러가 나. 여러 곳에서 들어야 하면 `.asBroadcastStream()` 으로 바꾸거나 broadcast controller 를 써.

### 3) 만드는 방법 3가지

**(a) async\* 와 yield — generator (가장 Dart 다움)**

```dart
// async* = "비동기 generator". yield 로 값을 하나씩 흘려보냄
Stream<int> countStream(int max) async* {
  for (int i = 1; i <= max; i++) {
    await Future.delayed(Duration(seconds: 1));  // 1초마다
    yield i;   // 값 하나 내보내기 (return 이 아니라 yield!)
  }
}
```

`yield` 는 `return` 과 달리 함수를 끝내지 않고 **값만 하나 흘려보내고 계속 진행**해. 루프가 끝나면 자동으로 `done` 이벤트 발생.

**(b) StreamController — 수동 제어 (가장 유연함)**

```dart
final controller = StreamController<int>();

// sink(입구)로 값을 밀어넣음
controller.sink.add(1);
controller.sink.add(2);
controller.sink.addError('문제 발생');
controller.close();   // done 이벤트

// stream(출구)으로 값을 받음
controller.stream.listen(
  (value) => print('받음: $value'),
  onError: (e) => print('에러: $e'),
  onDone: () => print('끝'),
);
```

`StreamController` 는 입구(`sink`)와 출구(`stream`)가 분리된 파이프야. BLoC 패턴의 심장이 바로 이거야.

**(c) 이미 만들어진 Stream 팩토리**

```dart
Stream.periodic(Duration(seconds: 1), (i) => i);  // 1초마다 0,1,2...
Stream.fromIterable([1, 2, 3]);                   // 리스트를 스트림으로
Stream.value(42);                                  // 값 1개짜리 스트림
```

### 4) 받는 방법 2가지

**(a) await for — 루프처럼 (읽기 쉬움)**

```dart
Future<void> main() async {
  await for (final value in countStream(3)) {   // 값이 올 때마다 루프 1회
    print(value);   // 1 ... 2 ... 3
  }
  print('완료');   // done 후 실행
}
```

**(b) listen — 콜백 등록 (제어 많음)**

```dart
final subscription = countStream(3).listen(
  (value) => print(value),
  onError: (e) => print('에러: $e'),
  onDone: () => print('완료'),
);

subscription.pause();    // 일시정지
subscription.resume();   // 재개
subscription.cancel();   // 구독 취소 (이거 안 하면 메모리 누수!)
```

`listen` 이 돌려주는 `StreamSubscription` 으로 pause/resume/cancel 을 제어해. **cancel 을 안 하면 메모리 누수**가 생겨 (주의점에서 다시).

### 5) 변환 — 흐르는 물을 가공하기

Stream 은 데이터가 흐르는 중간에 가공할 수 있어. List 의 메서드와 비슷:

```dart
countStream(5)
  .where((n) => n.isEven)      // 짝수만 통과
  .map((n) => n * 10)          // 10배로 변환
  .listen(print);              // 20, 40
```

이게 반응형 프로그래밍의 핵심 — 데이터 파이프라인을 선언적으로 조립해.

## 기본 예제 (단계별)

```dart
// 1단계: 가장 단순한 스트림
Stream<int> simple() async* {
  yield 1;
  yield 2;
  yield 3;
}

// 2단계: 시간 간격을 두고
Stream<int> timed() async* {
  for (int i = 1; i <= 3; i++) {
    await Future.delayed(Duration(seconds: 1));
    yield i;
  }
}

// 3단계: 변환 파이프라인
void main() async {
  await for (final n in timed().map((x) => x * x)) {
    print('제곱: $n');   // 1 ... 4 ... 9 (1초 간격)
  }
}
```

## Flutter에서는 어떻게 쓰이는가

여기가 Stream 이 빛나는 곳이야. Flutter 의 상태관리 상당수가 Stream 기반이야.

### 1) StreamBuilder — 스트림을 UI 에 직접 연결

```dart
StreamBuilder<int>(
  stream: counterStream,          // 들을 스트림
  initialData: 0,
  builder: (context, snapshot) {
    // 새 값이 올 때마다 이 builder 가 자동 재실행됨
    if (snapshot.hasError) return Text('에러: ${snapshot.error}');
    if (!snapshot.hasData) return CircularProgressIndicator();
    return Text('카운트: ${snapshot.data}');   // 값 표시
  },
)
```

스트림에 새 값이 흐를 때마다 `StreamBuilder` 가 **알아서 위젯을 다시 그려.** `setState` 를 직접 안 불러도 UI 가 데이터 흐름에 자동 반응해. (engine 편의 렌더링 파이프라인이 여기서 도는 거야.)

### 2) BLoC 패턴 — Stream 기반 상태관리의 정석

BLoC(Business Logic Component)는 "입력 이벤트 Stream → 비즈니스 로직 → 출력 상태 Stream" 구조야:

```dart
class CounterBloc {
  final _controller = StreamController<int>();
  int _count = 0;

  Stream<int> get count => _controller.stream;   // UI 가 구독할 출구

  void increment() {                              // UI 가 호출할 입구
    _count++;
    _controller.sink.add(_count);                 // 새 상태를 흘려보냄
  }

  void dispose() => _controller.close();          // 반드시 정리!
}
```

UI 는 `StreamBuilder` 로 `bloc.count` 를 듣고, 버튼은 `bloc.increment()` 를 불러. **UI 와 로직이 Stream 으로 분리**되는 게 BLoC 의 핵심 가치야. (Riverpod, GetX 등 더 현대적인 대안도 있지만 내부 원리엔 Stream 사상이 깔려 있어.)

### 3) Firebase, 웹소켓 등 실시간 데이터

```dart
// Firestore 문서 변화를 실시간 스트림으로
FirebaseFirestore.instance.collection('chat').snapshots()
// → 새 메시지가 추가될 때마다 Stream 이 새 값을 내보냄 → StreamBuilder 가 채팅 UI 갱신
```

실시간 채팅, 알림, 라이브 대시보드가 전부 이 패턴이야.

## 다른 언어와 비교

Stream 은 Dart 만의 발명이 아니라 **Reactive Streams** 라는 공통 사상의 Dart 구현이야:

| 언어/플랫폼 | 대응 개념 | 특징 |
|---|---|---|
| **JavaScript** | `AsyncIterator` / RxJS `Observable` | RxJS 가 사실상 표준. Dart Stream 과 가장 닮음 |
| **Kotlin** | `Flow` | 코루틴 기반. `emit` ≈ `yield`, `collect` ≈ `await for` |
| **Swift** | `Combine` 의 `Publisher`, `AsyncSequence` | Apple 생태계 반응형 |
| **Java** | `Reactive Streams` / RxJava | 안드로이드에서 많이 씀 |

**"왜 Dart 는 이렇게 선택했는가"**: Dart 는 Stream 을 **언어 레벨에 내장**했어 (`async*`, `yield`, `await for` 가 문법). RxJS 는 라이브러리고, Kotlin Flow 는 코루틴 위의 라이브러리야. Dart 는 비동기 단일 값(`Future` + `async`/`await`)과 비동기 다중 값(`Stream` + `async*`/`await for`)을 **대칭적인 문법으로** 언어에 박아넣어서 일관성을 줬어. `Future` 를 이해하면 `Stream` 은 "그것의 복수형" 으로 자연스럽게 확장돼.

| 단일 값 | 복수 값 |
|---|---|
| `Future<T>` | `Stream<T>` |
| `async` | `async*` |
| `return` | `yield` |
| `await` | `await for` |

이 대칭이 Dart 비동기 설계의 아름다운 점이야.

## 주의할 점

1. **구독 취소(cancel) 안 하면 메모리 누수.** `listen` 한 스트림은 위젯이 사라져도 계속 살아있어. Flutter 의 `StatefulWidget` 에서는 `dispose()` 에서 `subscription.cancel()` 과 `controller.close()` 를 **반드시** 해야 해. (`StreamBuilder` 는 자동으로 해주니 가능하면 그걸 써.)
2. **Single-subscription 스트림을 두 번 listen 하면 런타임 에러.** 여러 곳에서 들어야 하면 broadcast 스트림으로.
3. **`await for` 는 done 까지 그 자리에서 멈춰 기다린다.** 무한 스트림에 `await for` 를 쓰면 그 뒤 코드가 영원히 실행 안 돼. 무한 스트림은 `listen` 으로.
4. **에러도 흐른다.** Stream 의 에러는 try-catch 가 아니라 `onError` 콜백(또는 `await for` 를 try-catch 로 감싸기)으로 받아야 해.
5. **StreamController 의 sink 와 stream 을 혼동 주의.** `sink` 는 입구(값 넣기), `stream` 은 출구(값 받기). 방향이 반대.
6. **broadcast 는 "지나간 값" 을 못 받는다.** 늦게 구독하면 그 전에 흐른 값은 놓쳐. 마지막 값을 캐싱하려면 `BehaviorSubject`(rxdart 패키지) 같은 걸 써.
